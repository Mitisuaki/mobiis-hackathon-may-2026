# Plano: Integrar Proposta com API n8n (Reunião → Transcrição → Resumo)

## Contexto
A tela `Proposal` (`/dashboard/proposal`) atualmente exibe dados mockados hardcoded (3 negociações estáticas). O usuário deseja que a proposta seja gerada a partir de dados reais:
1. Selecionar uma reunião da API documentada em `N8N_INTEGRATION.md` (`http://204.216.160.72:3000`)
2. Obter o conteúdo da transcrição (`segments` ou `plainText`)
3. Enviar esse conteúdo como arquivo JSON para o webhook n8n: `https://danielsilva86.app.n8n.cloud/webhook-test/v1/gerar-resumo-reuniao`
4. Exibir o resumo/proposta retornado pelo n8n na tela

## Arquitetura do Fluxo

```
[Usuário seleciona reunião na tela Proposta]
    └──→ [MeetingService: GET /users/{id}/meetings]
           └──→ [MeetingService: GET .../transcripts]
                  └──→ [MeetingService: GET .../content?format=segments]
                         └──→ [ProposalService: POST multipart/form-data para webhook n8n]
                                └──→ [ProposalComponent exibe resumo retornado]
```

## Passos

### 1. Criar branch separada
```bash
git checkout -b feature/proposta-api-n8n
```

### 2. Habilitar HttpClient (se necessário)
- Verificar `src/app/app.config.ts`. Caso `provideHttpClient()` não esteja presente (o main atual não tem), adicioná-lo.

### 3. Atualizar ProposalService
- Arquivo: `src/app/core/services/proposal.service.ts`
- Refatorar para usar `HttpClient`:
  - `generateProposalFromTranscript(segments: TranscriptSegment[]): Observable<ProposalSummary>` — converte os segments em `Blob`/`File`, monta `FormData` com campo `file`, e faz `POST` para o webhook n8n.
  - Manter `downloadProposal()` para permitir download do resumo como `.docx` (placeholder).
- Tratamento de erro: se o webhook estiver offline (`webhook-test` só funciona quando o workflow está em "Listen for Test Event"), retornar mensagem amigável.

### 4. Criar interfaces/model para a resposta do webhook
- Arquivo: `src/app/core/models/proposal.model.ts` (novo)
- Interface `ProposalSummary`:
  ```ts
  interface ProposalSummary {
    resumo: string;
    problemas: string[];
    sentimento: string;
    acoesRecomendadas: string[];
    topicos: string[];
  }
  ```

### 5. Refatorar ProposalComponent
- Arquivo: `src/app/domains/dashboard/proposal/proposal.component.ts`
- Novo fluxo de UI:
  1. **Passo 1 — Selecionar Reunião**: lista de reuniões obtidas da API (`MeetingService.getMeetings()`). Filtro por assunto.
  2. **Passo 2 — Carregar Transcrição**: automaticamente busca transcrições da reunião selecionada e depois o conteúdo (`?format=segments`).
  3. **Passo 3 — Gerar Proposta**: botão "Gerar Proposta IA" que envia os `segments` para o webhook n8n.
  4. **Passo 4 — Exibir Resumo**: após resposta do webhook, renderiza o conteúdo da proposta (título, client, topics, etc. conforme retornado).
- Estado controlado por signals: `loadingMeetings`, `loadingTranscript`, `loadingProposal`, `selectedMeeting`, `transcriptContent`, `proposal`.
- Fallback: se o webhook falhar, exibir mensagem de erro e manter opção de re-tentar.

### 6. Atualizar i18n
- Adicionar chaves no namespace `PROPOSAL`:
  - `SELECT_MEETING`, `GENERATE_PROPOSAL`, `GENERATING`, `NO_TRANSCRIPT`, `ERROR_WEBHOOK`, `RETRY`

### 7. Estilos
- Ajustar `proposal.component.css` para acomodar o novo fluxo de 3 passos (seleção → geração → visualização).

### 8. Validação
- `tsc --noEmit` para garantir tipagem correta.
- Teste manual: selecionar reunião, gerar proposta, verificar exibição.
